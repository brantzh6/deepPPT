# Validated Slide Patterns From This Project

## Purpose
This file captures the slide patterns that proved useful during the OpenClaw executive deck work.
These are not abstract ideas only. They are concrete reusable page types with observed strengths.

## Pattern 1. Cover Orbit
ID:
- `cover_orbit`

Use for:
- executive cover
- strategic topic framing
- technology transition theme

Visual structure:
- large left title block
- right orbit/core object
- secondary signal or story blocks

Why it works:
- immediate focal point
- strong “technology system” feel
- supports strategic framing instead of plain title-only cover

Do not use for:
- highly detailed content pages
- lists or architecture-heavy pages

## Pattern 2. Narrative Map Agenda
ID:
- `narrative_map`

Use for:
- deck agenda
- chapter framing
- “three-part story” introduction

Visual structure:
- one dominant chapter card
- one or two supporting chapter cards
- supporting concept chips / decision cue

Why it works:
- avoids dead three-column summary look
- gives the deck a visible narrative hierarchy

Do not use for:
- pages with many equal items

## Pattern 3. Trust Terminal
ID:
- `trust_terminal`

Use for:
- explaining why a system feels trustworthy
- showing architecture from reasoning to action
- demonstrating tangibility

Visual structure:
- conceptual explanation block on one side
- terminal or layered runtime block on the other side

Why it works:
- strong object anchor
- makes abstract trust argument feel concrete

Do not use for:
- pages without execution / feedback logic

## Pattern 4. Closed Loop Process
ID:
- `closed_loop_flow`

Use for:
- execution loops
- generate / execute / verify / fix flows
- self-correction logic

Visual structure:
- ordered loop or process stages
- distinction between bad loop and good loop if needed

Why it works:
- clarifies why agents are different from plain generation systems

## Pattern 5. Bottleneck Shift
ID:
- `bottleneck_shift`

Use for:
- explaining paradigm shifts in engineering
- “write slower -> review better” idea
- role transition under AI

Visual structure:
- large left statement
- right support cards
- bottom role / criteria blocks

Why it works:
- one sharp thesis
- supporting evidence without losing drama

Do not use for:
- neutral information pages

## Pattern 6. Skill Reconstruction
ID:
- `skill_reconstruction`

Use for:
- changing value of human skills
- appreciating vs depreciating capability split

Visual structure:
- left side: appreciating capabilities
- right side: depreciating capabilities

Why it works:
- clean executive comparison
- easy to adapt to multiple topics

## Pattern 7. Human-AI Relationship Evolution
ID:
- `evolution_split`

Use for:
- relationship evolution
- assistant -> copilot -> controlled agent
- digital employee framing

Visual structure:
- left current/legacy state
- center direction arrow
- right future but controlled state
- bottom principles block

Why it works:
- supports gradual evolution, not hype
- useful for enterprise realism

## Pattern 8. Product Matrix Trio
ID:
- `product_trio_matrix`

Use for:
- product family comparison
- vendor portfolio explanation
- “how to choose” discussions

Visual structure:
- top positioning strip
- three differentiated product cards
- bottom decision guidance strip

Why it works:
- avoids plain table
- supports both differentiation and selection

Do not use for:
- large numbers of products

## Pattern 9. Layered Architecture
ID:
- `layered_architecture_stack`

Use for:
- enterprise architecture explanation
- runtime/model/data/security decomposition
- platform stack discussions

Visual structure:
- stacked or stratified layers
- side panel for control logic or deployment note

Why it works:
- makes enterprise platform logic visible
- better than tables for stack explanation

## Pattern 10. Runtime Pooling
ID:
- `runtime_pool_map`

Use for:
- mapping workloads to environments
- infrastructure choice pages

Visual structure:
- environment sequence or pool structure
- side explanation for governance logic

Why it works:
- turns infrastructure into a decision model

## Pattern 11. Scenario Flow
ID:
- `scenario_flow`

Use for:
- scenario A / B pages
- user workflow
- operational loops

Visual structure:
- process track on one side
- outcome / risk / fit blocks on the other

Why it works:
- makes scenarios actionable
- avoids pure bullet scenario pages

## Pattern 12. Decision Frame
ID:
- `decision_frame`

Use for:
- CIO / CxO summary
- deployment and governance criteria
- landing decision discussion

Visual structure:
- structured decision questions
- grouped concerns
- emphasis on boundaries and accountability

Why it works:
- executive-friendly
- turns product capability into a governance conversation

## Pattern 13. Risk Split / Double Helix
ID:
- `risk_split`

Use for:
- capability vs risk discussion
- control-first framing
- executive risk opening

Visual structure:
- left positive capability side
- right negative risk side
- central coupling marker

Why it works:
- sharp strategic contrast
- very reusable in enterprise AI topics

## Pattern 14. Amplification Effect
ID:
- `amplification_chain`

Use for:
- small error -> large consequence explanation
- misconfiguration or automation scaling risks

Visual structure:
- input / loop / output chain
- increasing severity from left to right

Why it works:
- communicates risk mechanics, not only warnings

## Pattern 15. Minimum Self-Protection
ID:
- `principles_triptych`

Use for:
- action principles
- minimum safeguards
- baseline governance checklist

Visual structure:
- 3 large principle blocks
- distinct color accents

Why it works:
- compact and memorable
- good for action guidance

## Pattern 16. Security Control Plane
ID:
- `security_control_plane`

Use for:
- enterprise security center pages
- multi-domain governance architecture

Visual structure:
- top policy/governance bar
- four or more capability towers
- bottom foundation strip

Why it works:
- reads like a real security architecture, not a table

## Pattern 17. Chapter Summary
ID:
- `chapter_summary_signal`

Use for:
- end of chapter synthesis
- one dominant takeaway + decision implication

Visual structure:
- dominant summary block
- secondary implication block
- decision cue or signal block

Why it works:
- closes a chapter cleanly
- useful for executive decks

## Pattern 18. Closing Control-First
ID:
- `closing_control_first`

Use for:
- deck ending
- high-level call to action
- strategic summary

Visual structure:
- left year / final thesis
- right core orbit / control markers

Why it works:
- memorable and concise
- supports executive call to action

## Recommended First Pattern Set For MVP
If only a small page library can be implemented first, start with:
- cover_orbit
- narrative_map
- trust_terminal
- closed_loop_flow
- bottleneck_shift
- evolution_split
- layered_architecture_stack
- scenario_flow
- risk_split
- security_control_plane
- chapter_summary_signal
- closing_control_first
