# Design Discussion Memory

## Purpose
This file preserves the design-specific parts of the conversation in a concentrated form so that future work does not lose the reasoning that emerged from repeated critique.

## The Most Important Design Point
The user repeatedly established that:
- style is not a cosmetic layer
- style is part of the argument
- a weak page damages credibility, even if the content is correct

This means future deck generation must treat visual expression as part of meaning, not post-processing.

## What “Good” Meant In This Project
The target standard for strong slides included:
- deliberate asymmetry
- visible focal contrast
- strong highlighted phrases
- layered gradients and glows
- icon and status-pill systems
- terminal-like UI motifs
- composition with object anchors
- enough density to feel substantial, but not crowded

The user specifically rejected:
- same-size repetitive cards
- large empty dark areas
- pages where text sat only in the upper region
- plain tables used as architecture pages
- weak timeline layouts
- “AI slop” layouts that looked generic

## Key Discussion Themes About Style

### 1. A Page Must Feel Intentional
A strong page should look like:
- it was designed around the claim
- not like content was poured into a generic shell

### 2. Composition Is More Important Than Decoration
The user repeatedly pointed out that:
- correct color palette was insufficient
- glow alone was insufficient
- structure had to match the point of the slide

### 3. One Page, One Dominant Move
The strongest pages usually expressed one clear move:
- define
- contrast
- stage a transition
- show a control center
- visualize a bottleneck
- expose a risk structure

### 4. Layout Must Fit Content Role
Examples from the discussion:
- a timeline page should use a real axis or staged progression
- a trust page should use architecture and execution objects
- a chapter summary should not be a dead three-column page
- a product choice page should help selection, not just compare specifications

### 5. Empty Space Must Be Managed, Not Ignored
The user repeatedly called out:
- lower halves left empty
- content packed at the top
- large containers with too little inside

This became a hard lesson:
- visual emptiness must be treated as a layout bug

## Specific Patterns The User Valued In Reference Slides
- cover with relationship map or orbit
- left-right split where one side is subdued and the other is activated
- giant thesis phrase with smaller structured support blocks
- terminal-like panels with execution grammar
- layered architecture cards with subtle glow and depth
- risk pages with stronger red emphasis and mechanical framing
- chapter ending pages with one clear strategic judgment

## Specific Patterns The User Rejected
- equal-width repeated boxes
- plain card matrices as final expression
- chapter pages that looked like tables
- rich color but flat structure
- timeline content without visual timeline
- verbose text panels without hierarchy

## Design Memory For Future Systems
The future system should explicitly preserve:
- page weight center
- object anchor choice
- asymmetry rules
- density rules
- hierarchy rules
- chapter-level style logic

These should be encoded as part of:
- pattern cards
- page-type rules
- theme systems
- QA checks

## Recommended Future Rule
Every page should answer these five design questions:
1. What is the narrative role of this page?
2. What is the dominant object or focal anchor?
3. Where is the visual weight center?
4. What is intentionally de-emphasized?
5. Why does this page look different from a generic template?

If a page cannot answer these questions, it is likely not ready.
