# Style Lessons From This Project

## Why This File Exists
This file records the concrete style lessons learned while producing the OpenClaw executive deck.
It is not generic design theory. It captures the patterns that repeatedly improved quality, and the anti-patterns that repeatedly made pages look weak.

## Core Observation
Visual quality problems were rarely caused by color alone.
They were usually caused by:
- wrong composition
- weak page hierarchy
- lack of a visual anchor
- empty lower half of the page
- generic card repetition
- mismatch between content role and page type

## Practical Style Rules

### 1. Color Match Is Not Enough
Matching reference colors does not produce a premium page.
The following also matter:
- asymmetry
- focal contrast
- object-centric layout
- rhythm across the page
- text-to-graphic balance

### 2. Every Important Page Needs A Visual Anchor
Strong pages had a dominant object, not only text boxes.

Examples:
- terminal mockup
- orbit / core node
- timeline axis
- layered architecture stack
- central decision node
- signal chart / bars

If a page had no anchor, it usually looked like a low-end template.

### 3. Page Weight Must Be Designed Explicitly
One recurring failure mode was top-heavy composition:
- title and text packed at the top
- large empty lower region

This made many pages feel unfinished.

For each page, decide the intended weight center:
- upper-weighted: only for dramatic openers
- middle-weighted: best for architecture and comparison
- lower-weighted: useful when the page needs grounding or a large hero object

### 4. Avoid Repeating Generic Card Grids
Three or four equal cards on every page quickly flatten the deck.

What works better:
- one dominant area + one supporting area
- left-right contrast
- center axis with asymmetric blocks
- hero object + supporting metrics
- one large concept card + two smaller control cards

### 5. Strong Enterprise Pages Need “Decision Objects”
For executive decks, the page should look like it is helping a decision, not just displaying information.

Useful decision objects:
- control core
- risk split
- governance stack
- timeline with checkpoints
- score / signal bars
- choice matrix

### 6. “Tech Feel” Is Not Just Glow
Technology style is not achieved by glow alone.

The pages that felt strongest used:
- terminal grammar
- chips / status pills
- layered frames
- subtle grid backgrounds
- controlled gradients
- disciplined typography
- strategic use of red / cyan / orange accents

### 7. Empty Boxes Must Be Treated As Bugs
Several pages degraded because a large container had little content.

Rule:
- if a box occupies meaningful page area, it must contain either
  - a visual object
  - a structured group of points
  - a comparative statement
  - or a meaningful metric

An empty box is not neutral. It lowers perceived quality.

### 8. Good Pages Usually Express One Strong Move
The best pages from this project did not try to show everything.
They usually did one clear thing:
- define a concept
- compare two paradigms
- explain why something feels trustworthy
- show a bottleneck shift
- frame a decision
- expose a risk structure

## Common Anti-Patterns Observed

### Anti-Pattern 1. Symmetry Without Purpose
Perfect symmetry often made pages look generic.
Use symmetry only when the page is explicitly about equivalence or comparison.

### Anti-Pattern 2. Text Dump In Dark Theme
Dark theme makes weak composition more obvious.
Long bullets inside large dark cards feel heavy and amateur unless broken by objects or hierarchy.

### Anti-Pattern 3. Generic Agenda Cards
A three-column chapter summary page is usually too weak for executive decks.
Agenda pages need stronger narrative staging:
- dominant chapter
- secondary chapter
- supporting decision cue
- visible theme chips or concept markers

### Anti-Pattern 4. Wrong Scale Hierarchy
If the title is huge but everything else is tiny and disconnected, the page looks hollow.
Scale must connect title, supporting statement, and core object.

### Anti-Pattern 5. Fake “Richness”
Adding more boxes does not make a page richer.
Richer pages come from:
- layered information density
- visual relationships
- stronger anchors
- contrast between primary and secondary content

## What Worked Best In This Deck

### Worked Well
- cover with orbit/core concept
- trust page with terminal stack
- bottleneck shift page with giant left statement and structured right support
- chapter summary pages with one dominant claim and one decision block
- security center page when treated as architecture control plane, not plain matrix
- product selection page when framed as choice + positioning + boundaries

### Worked Poorly Until Fixed
- plain equal-width card pages
- top-heavy chapter maps
- pages with large empty lower containers
- scenario pages with no process object
- safety pages that looked like tables instead of control systems

## Guidance For Future Decks
- choose page type by narrative role, not by convenience
- place at least one anchor object on every important slide
- treat visual emptiness as a quality defect
- vary internal composition within a consistent theme
- use model judgment to decide expression form, not only wording
